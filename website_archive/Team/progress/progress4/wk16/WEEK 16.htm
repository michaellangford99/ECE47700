After this was done, I finished the initial implementation of the overall LiDAR code. This was implemented as a state machine based off some relatively simple concepts. These included the desired travelling along the distance of a wall, calibrating against the wall, recalibrating if straying off track from the wall, the initial positioning (including calibration) to the first wall, changing walls (including increasing set distance from wall if traversed around all walls of the room), and lastly, a stop state if an object was detected. The speed of the drone was based on a simple base set angle and scaled based on a safety padding distance to the wall. This was also exponentially decayed to only have the drone move as the change in angle is acceleration and not a velocity, so we only want the drone to move in small increments. Michael helped me remember that fact as well as the fact that having the motors go from 100 to 0 quickly is pretty dangerous, so I tried to add a filter to reduce this effect, however I didn’t seem to calibrate it to have a big enough impact in the final showcase.

This led into making the Pi Zero easier to communicate with as debugging using the method of removing the SD card from the Pi Zero, changing config settings to allow keyboard and mouse use, then moving it to another Pi was becoming unacceptable. So, I had to enable USB over ethernet SSH on the Pi Zero (more specifically on the kernel that was installed on the SD Card). I encountered numerous problems when trying to enable this, such as the SSH server being active on the Pi 3 and Pi 2 but not working on the Pi Zero, the Pi Zero SSH server being active but refusing connections, the Pi Zero refusing to create and IPv4 static IP, and some others that I am unable to remember at this moment. Long story short, after reinstalling OpenSSH, updating it and doing a few other things, I found a forum from what I would guess is the late 2000’s at the latest, about similar issues arising on a Linux Distro running MongoDB and a few kernel lines that needed changing. I honestly have no idea what I changed but I was eventually able to SSH with USB over ethernet into the Pi as shown below in Figure 3.

Figure 3: SSH with USB over ethernet into the Pi Zero mounted on the drone (under the flight controller)

After this, I worked with Michael to make sure that the Pi was communicating properly with our flight controller. We created a custom package to send all of the LiDAR data, the camera state and the yaw, pitch and roll commands from the Pi to the flight controller. The package had a 1-byte start bit, 65 (ASCII A), a string representation of whether or not the camera was ready for image recognition, the distance in mm of each of the 4 LiDAR sensors, and then the yaw, pitch, and roll commands in desired degree change (acting almost as a replacement controller) and finally a 1-byte end bit, 66 (ASCII B). This package, name packed, is shown in Figure 4 below.

Figure 4: The package being sent to the Pi

Next, Jon and I started on the LiDAR mounting mechanisms. This entailed a few things, first, we had to mount the LiDARs to the 3d printed mounts that Jon had made over the past week. We also had to make a wiring harness for the 4 LiDAR sensors. This would be a BIG pain point over the week, with multiple hours trying and failing to get a proper wiring harness together. This was such an issue as we wanted as little solder points on the flight controller as possible. This arose as the I2C pins on the pi were directly next to the pins that were being used for UART communication. This meant that the connector from the Pi had BOTH the I2C pins and the UART pins. This meant 1 of 2 things, either have a cascading connection, or connect all the LiDARs together in parallel. Both were tried but after numerous failed attempts, the cascading harness was made by connecting one end to the front of the LiDAR and having is cascade to the back of the next LiDAR (Thank you Michael and Santiago for this one, mine was poorly designed/put “together”). Luckily, the XSHUT connectors were much easier to assemble on the board. This connector can be seen blow in Figure 5.

Figure 5: The LiDAR harness can be seen as the Red, Black, Green and Blue connector that is connected to the front and back of the LiDAR

After this I had to implement Jon’s object detection code. This required a bit of tweaking on my end with the exact values used to determine if an object would be detected when flying. This wasn’t too bad as the code Jon had written already had sliders to retest what values could be used. Turns out we found very solid color mask values previously. These values are shown in Figure 6 but can also be found in one of my previous progress reports.

Figure 6: The previously found mask values

This added quite a lot of other problems, however. First, the code that Jon created found an object by calculating the mean of the mask’s pixels and the value where an object was detected was 49. However, as the drone would be flying and prone to a lot of vibration, as well as having the drone be relatively far away, I found that the most reliable value to detect and object was ~10. This is the value that was used in the final design. One of the other problems was that, while it wasn’t taking too much CPU power to get and process an image, it was halting the Pi’s ability to continue to read and send LiDAR and navigation data to the flight controller. To solve this, I had to “multithread” the camera processes along with the navigation calculations. Multithread is in quotes as all of this was done in Python and is much better described as “running multiple processes asynchronously while still being on the same compute thread”. While multithreading in Python I did see a drop in performance, but it didn’t affect the amount of navigation commands being sent to the Pi, only the number of frames being taken per second. Luckily, it still all seemed to work well. This came into play mainly for the demonstration of the drone detecting an image. I had to change the camReady string variable that was in the package data shown above to something that could indicate that the drone actually spotted an object. This was done by making the camReady variable controlled by a thread Event class. This allowed me to set a True or False statement that would carry between the two threads based on whether the camera had detected an object. If it did, the camera thread would set the event to True and the camReady variable would change from a 0 to a 1. Michael then implemented that package (1 was sent as an ASCII 1 so it was sending a 48 when not detected and 49 when it was detected) to check when that character in the package changed to the 1 and then had it start a landing sequence by scaling down the auto throttle target altitude down by 0.996f every loop cycle of the flight controller. This was how we ended up demonstrating the camera integration onto the device.

 

I also added logging for both the pictures being added and the IMU yaw pitch and roll values that the Pi was receiving from the flight controller. Both the text log files and the pictures were stored very similarly. They were always stored in the /root/ drive of the Pi, as that is where the python process for running the code at boot was operating. The log files were stored with the filenames, “log_YEAR-MONTH-DAY_HOURS-MINUTES-SECONDS.txt”, and the pictures were stored with the filenames being, “img_YEAR-MONTH-DAY_HOURS-MINUTES-SECONDS.jpg”. I changed to this from the original filenames which used colons for the hour:minutes:seconds as it made it easier for the files to be WinSCP off of the Pi, dude to Windows hate for semicolons in filenames.
